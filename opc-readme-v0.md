一、文档基础信息

项目名称

weopc.org 平台

产品版本

MVP 1.0（冷启动版本）

产品定位

聚焦OPC/一人公司，以「商业搜索为核心、项目供需对接为主体、工具箱为辅助、稳健创业价值观为底色」的精准撮合平台，核心解决OPC找靠谱项目、项目方招精准人才/资源的痛点，同时兑现对嘉宾的感恩回报





目标用户

1.  项目方：已跑通0-100的OPC嘉宾、优质OPC创业者（核心需求：曝光项目、招募人才/合伙人、对接资源、渠道管理）；2.  需求方：找项目的OPC创业者、独立开发者、轻创业意向者（核心需求：找已验证项目、对接项目方、获取报价）





核心目标

1.  冷启动期：完成核心功能落地，积累20-30个优质项目、500+精准客资，实现30+组项目对接；2.  成长期：搭建完整的付费变现体系，实现工具+项目对接双重变现，成为OPC稳健创业核心撮合平台





文档目的

明确平台MVP版本核心功能、交互逻辑、落地优先级，为开发、运营提供统一参考，确保功能贴合前期运营需求，轻量落地、快速闭环；同时基于前后端分离架构，预留多端拓展空间，适配后续网页端等形态的迭代。





关联需求

前期运营规划（项目招商、客资挖掘、嘉宾对接）、BOSS直聘式变现逻辑、大众点评式点评体系、链接中转工具箱需求、前后端分离架构适配需求、多端（小程序+网页端）兼容需求、三角色权限管控需求





产品形态

采用「前后端分离架构」，MVP 1.0优先落地移动端小程序（核心冷启动载体），预留网页端拓展空间，后续逐步迭代网页端，实现多端数据互通、体验统一，具体形态规划如下：1.  MVP 1.0：聚焦小程序，快速触达目标用户（OPC移动办公群体），完成冷启动核心目标；2.  短期迭代（MVP上线后1-2个月）：基于同一套后端服务，开发网页端，适配PC端办公场景（如运营后台网页版、项目方批量操作网页版）；3.  长期迭代：实现小程序+网页端全功能同步、数据实时互通，满足不同用户场景需求（移动端碎片化使用、PC端高效操作）。





核心角色定义

平台核心分为三大角色，适配「用户使用-商家运营-平台管理」的完整闭环，基于前后端分离架构，实现三角色多端权限适配，具体定义如下：1.  普通用户角色（类比淘宝用户）：平台需求方主体，核心用于找项目、对接项目方、使用工具箱、提交点评；2.  项目提供方角色（类比淘宝商家）：平台供给方主体，核心用于入驻平台、发布项目、管理项目信息、对接需求方、查看客资；3.  后台管理员角色：平台管控主体，核心用于审核项目、管理客资、维护点评体系、配置平台规则、管控其他两角色权限。





二、核心产品逻辑

核心逻辑：以嘉宾优质项目为种子，以商业搜索为入口，串联「项目供需对接+大众点评式信任背书+链接中转工具辅助」三大核心模块，采用前后端分离架构，MVP 1.0优先落地小程序快速冷启动，预留网页端拓展空间，前期靠免费功能积累精准流量，后期复刻BOSS直聘「项目方付费提权+增值服务」的变现模式，同时渗透「先做牛马后做爷、稳健创业」的价值观，形成「工具引流→项目对接→点评信任→付费变现」的闭环，实现平台、嘉宾、用户三方共赢，兼顾当下冷启动效率与长期多端拓展。

简化逻辑链：普通用户（需求方）→ 小程序搜索首页找项目→ 查看项目详情（报价+落地成果+点评）→ 留资/对接项目提供方→ 对接后发布点评→ 项目提供方（嘉宾/优质项目）→ 小程序/运营后台（后续网页端）入驻填项目/报价→ 免费获取精准客资→ 付费提权放大曝光→ 工具箱（链接中转/域名工具）→ 普通用户免费使用引流→ 联动项目页实现流量转化→ 增值服务变现→ 后台管理员→ 小程序/网页端后台审核项目、管理客资、维护平台秩序→ 后期网页端上线，同步所有核心功能及三角色权限，适配PC端高效操作场景。

2.1 前后端分离架构核心要求（适配多端+三角色）

为保障后续网页端顺利拓展及三角色权限管控，MVP 1.0开发阶段需遵循以下架构要求，确保多端复用、高效迭代、权限清晰：1.  后端：采用标准化接口设计，提供统一的数据服务，接口兼容小程序、网页端，同时区分三角色接口权限，避免越权操作，避免后续网页端开发重复开发后端逻辑；2.  前端：小程序与后续网页端共用一套核心业务逻辑，差异化适配各端交互体验（如小程序适配移动端单手操作、网页端适配PC端多窗口操作），同时针对三角色设计专属操作界面（普通用户侧重使用、项目提供方侧重管理、后台管理员侧重管控），保持视觉调性、核心功能一致；3.  数据：用户数据、项目数据、客资数据、点评数据、工具使用数据等，统一存储、实时同步，按角色权限分配数据查看/操作权限，确保小程序与网页端用户切换使用时，体验无缝衔接，权限同步；4.  运营后台：MVP 1.0可采用简易网页版（适配前期1-2人后台管理员运营），后续基于同一后端，优化网页端运营后台，强化三角色权限管控，实现更高效的批量操作、数据统计功能。

2.2 三大角色核心权限与功能适配（MVP 1.0）

核心原则：权限最小化、操作轻量化，贴合MVP冷启动需求，仅开放各角色核心必要权限，后期迭代逐步丰富，同时适配多端（小程序+后续网页端），确保角色操作与端侧场景匹配。

## 开发规范 / 常见 CI 问题记录

> 每次 push 前务必本地跑完整 build：`pnpm -r --filter=@weopc/api --filter=@weopc/web build`

### 1. Prisma schema — 新增 model 必须补双向关联

**问题**：新增含 `userId` 字段的 model（如 `Link`），只在 `User` 上加了 `links Link[]`，但 `Link` model 里缺少反向关联字段，导致 `prisma generate` 报 P1012 错误。

**规范**：每个 model 的外键字段都必须配一个对应的 `@relation` 字段。

```prisma
// ✅ 正确
model Link {
  userId  String
  user    User   @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

验证命令：`pnpm --filter @weopc/api exec prisma validate`

---

### 2. Express Router — 必须显式声明类型

**问题**：`const router = Router()` 推断类型引用了 pnpm 内部路径，TypeScript 编译报 TS2742 不可导出错误。

**规范**：所有路由文件统一加显式类型注解。

```typescript
// ✅ 正确
import { Router } from 'express';
const router: Router = Router();
```

---

### 3. Web 应用 — 禁止直接连接数据库

**问题**：`apps/web` 是 Next.js 前端应用，没有配置数据库连接。在页面文件中 `import { prisma } from '@/lib/database/prisma'` 会导致 build 失败（模块不存在）。

**规范**：Web 应用的所有数据读写必须通过 API（`NEXT_PUBLIC_API_URL`）进行，不得直接使用 Prisma。

```typescript
// ❌ 错误 — web 页面里直接查 DB
import { prisma } from '@/lib/database/prisma';
const link = await prisma.link.findUnique(...);

// ✅ 正确 — 通过 API 获取，或 client component 跳转到 API 端点
window.location.href = `${process.env.NEXT_PUBLIC_API_URL}/api/v1/links/t/${shortCode}`;
```

---

### 4. Web 应用 — 认证状态使用 useAuthStore

**问题**：`@/lib/hooks/useAuth` 不存在，使用该路径导入会导致 build 失败。

**规范**：认证状态统一从 Zustand store 获取。

```typescript
// ❌ 错误
import { useAuth } from '@/lib/hooks/useAuth';
const { user, isAuthenticated } = useAuth();

// ✅ 正确
import { useAuthStore } from '@/lib/store/auth';
const user = useAuthStore(state => state.user);
const isAuthenticated = useAuthStore(state => !!state.token);
```
